3. Extension Background

All hData section documents identified by an XML namespace must be declared in the extensions node of the root document. Extensions are either "mandatory" or "optional." Each section document has an address expressed as a Uniform Resource Identifier (URI). Each URI is registered in the extensions node of the root document. Sections can contain either single data type documents (e.g., XML, JSON, HTML) or URIs referencing other sections. Section document content must be expressed in XML and document types must be declared in the extensions node of the root document.

4. Description

4.1. Purpose

This is the information to be sent from one healthcare provider to another regarding a specific patient. The referral request may contain specific medical procedures to be performed (accompanied by previously obtained authorizations) and links to relevant clinical information.

4.2. Description

NameTypeMandatory/OptionalNotes
id string M Generated unique id for the referral request
date dateTime M The date and time the referral request was created
disposition code O

Contains the type of response or action that the requesting provider would like from the consulting provider.
number string O Contains unique permanent referral number assigned by the originating system.
priority code O Contains the urgency of the referral
    The valid values are:
  • S -- STAT
  • A -- ASAP
  • R -- Routine
reason URL O Link to the Condition entry that is the reason for this referral or consultation
status code O Contains the status of the referral.
requestedProcedure URL O Link to the Procedure entry that is requested by this referral or consultation.
reasonDescription string M Contains narrative text describing reason for referral.
category code O Contains the location at which the referral will take place.
  • I -- Inpatient
  • O -- Outpatient
  • A -- Ambulatory
  • E -- Emergency
requestedNumTreatments integer O Contains the requested number of times that the treatment may be administered to the patient without obtaining additional authorization
requestedSpecialty string O The requested specialty of the consulting provider
requestedConsultingProvider URL O Link to healththcare provider entry for the requested consulting provider
requestingProvider URL M Link to the healthcare provider who is requesting the referral or consultation
recommendedPlanOfCare URL O Link to a Plan of Care entry
authorizationEffectiveDate interval O Contains the effective interval of the authorization
authorizationID string O Contains the coverage application's permanent identifier assigned to track the authorization and all related billing documents
authorizationReimbursementLimit quantity O Contains the dollar limit for reimbursement specified by the coverage plan for the authorized treatment.
authorizationNumberOfTreatments quantity O Contains the number of times that the authorized treatment may be administered to the patient without obtaining additional authorization.
authorizationProcessDate dateTime O Contains the date that the authorization originated with the authorizing party
insurancePayor URL O Insurance provider information associated with the requested referral

4.3. Clinical Requirement

Referral Request is characterized by:

  • Reason for referral
  • Authorization information

4.4. XML Schema Definition (XSD)

The XML Schema Definition (XSD) for Referral Request specifies how to formally describe the elements in an XML document.

ReferralRequest.xsd is located at the following URI: https://raw.github.com/ssayer/health-data-standards/develop/greenc32/schemas/rest_plus/referralrequest.xsd

ReferralRequest.xsd uses complex data types defined in the data_types schema at the following URI: https://raw.github.com/ssayer/health-data-standards/develop/greenc32/schemas/rest_plus/data_types.xsd

4.5. Resource Representations

Referral Request may be represented in XML and JSON format.
